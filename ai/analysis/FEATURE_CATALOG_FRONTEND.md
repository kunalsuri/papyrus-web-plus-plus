## §0 Shape of the frontend

Two npm workspaces (`frontend/package.json`, turbo, Node 22.16.0):

- **`frontend/papyrus-web/`** (`@eclipse-papyrus/papyrus-web` 2026.5.0) — an
  11-source-file shell around the external `@eclipse-sirius/sirius-web-application`
  2026.5.0. ALL heavy UI (workbench, diagrams, forms, trees, tables) is external;
  the shell only contributes branding + the Profiles page via extension points.
- **`frontend/papyrus-web-components/`** (`@eclipse-papyrus/papyrus-web-components`
  2026.5.0) — the actual Papyrus UI library: custom nodes, custom widgets, profile
  modals, library commands, published to npm and consumed as a dependency
  (locally via yalc `publish:local`).

Everything is wired through ONE file: `frontend/papyrus-web/src/index.tsx` registers
Footer, nav-bar icon, help URL, Profiles menu item + routes into
`papyrusWebExtensionRegistry`, then renders `SiriusWebApplication` with
`nodeTypeRegistry={papyrusNodeTypeRegistryValue}`.

## §2 State Management & Hooks  `[inferred]`

No global state of its own — Apollo GraphQL client and workbench state live inside the
external Sirius packages. Papyrus-specific state:

| Mechanism | Path | Owns |
|---|---|---|
| xstate machine | `frontend/papyrus-web-components/src/profile/apply-profile/ApplyProfileModalMachine.tsx` | Apply-profile modal flow (xstate 4) |
| xstate machine | `frontend/papyrus-web-components/src/profile/apply-stereotype/ApplyStereotypeModalMachine.ts` | Apply-stereotype modal flow |
| GraphQL document transforms | `frontend/papyrus-web-components/src/nodes/NodesDocumentTransform.ts`, `frontend/papyrus-web-components/src/widgets/CustomWidgetsDocumentTransform.ts` | Inject Papyrus fragments into Sirius queries — the mechanism that makes custom nodes/widgets data flow without forking Sirius queries |
| Extension registry | `frontend/papyrus-web-components/src/registry/PapyrusWebExtensionRegistry.tsx` | All UI contribution wiring |
